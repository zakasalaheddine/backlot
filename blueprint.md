# backlot — Plugin Blueprint

A Claude Code plugin that builds a UGC content engine: **reusable characters**, **ad images**, and **UGC videos**, all sharing one continuity-locked asset library.

- **Images / compositing:** Nano-Banana (Gemini 2.5 Flash Image)
- **Video:** Seedance (image-to-video)
- **Planning, copy, orchestration, review:** Claude

> This is the reference doc for the repo. Build against it; adjust names to taste. Open decisions are flagged at the end — resolve them before the runner is written.

---

## 1. Core principle: the Character is the atomic asset

Everything the plugin makes is either **a reusable asset** or **an output built from assets**.

Two asset classes are persisted and reused forever:

- **Characters** — a UGC "creator" identity: locked reference images (face from multiple angles, body, signature look) + a text character sheet. Created once, reused across every ad and every video. This is what makes an account feel like a real person.
- **Products** — the real product: locked reference images + constraints (e.g. "print area text must stay legible").

Every output — an ad image, a video shot — is generated by **compositing locked references into a new scene**, never by regenerating the identity from scratch. Regeneration is where consistency dies. Reference-locking is the whole trick.

```
                 ┌─────────────────┐
                 │  ASSET LIBRARY  │
                 │  characters/    │  ← created once, locked, versioned
                 │  products/      │
                 └────────┬────────┘
                          │ referenced by ID
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
   character-creator   ad-image        ugc-video
   (produces assets)  (static ads)   (image→video)
```

## 2. Plugin structure

Follows the router + sub-skills pattern (same as `lincx-reports`). Shared prompt-craft and scripts live once and are referenced by every sub-skill.

```
backlot/
├── .claude-plugin/
│   └── plugin.json                 # name, version, description
├── commands/                       # optional slash commands
│   ├── ugc-character.md            # /ugc-character
│   ├── ugc-ad.md                   # /ugc-ad
│   ├── ugc-video.md                # /ugc-video
│   └── ugc-assets.md               # /ugc-assets  (list/inspect library)
├── skills/
│   ├── backlot/SKILL.md         # ROUTER — dispatches to sub-skills
│   ├── character-creator/SKILL.md
│   ├── ad-image/SKILL.md
│   ├── ugc-video/SKILL.md
│   └── _shared/                    # the real IP: prompt-craft
│       ├── continuity.md           # keeping a character identical across gens
│       ├── nano-banana.md          # compositing / editing prompt patterns
│       ├── seedance-motion.md      # motion prompts that don't drift
│       ├── ad-layout.md            # composition, overlay hierarchy, CTA
│       └── asset-library.md        # the on-disk conventions below
├── providers/                      # thin model-host abstraction (see §7)
│   ├── images.py                   # generate_reference(), composite()
│   ├── video.py                    # image_to_video()
│   └── llm.py                      # plan(), copy()
├── scripts/
│   ├── assets.py                   # create / list / get characters & products
│   ├── run_ad.py                   # ad job JSON → images (+ text overlay)
│   ├── run_video.py                # production JSON → refs → keyframes → clips → assemble
│   ├── render_shot.py              # regenerate ONE shot/variant by id
│   └── compose.py                  # ffmpeg concat + captions + audio
└── assets/                         # the persisted library (gitignored or DVC'd)
    ├── characters/
    │   └── <char-id>/
    │       ├── character.json
    │       └── refs/*.png
    └── products/
        └── <product-id>/
            ├── product.json
            └── refs/*.png
```

**Router responsibility** (`skills/backlot/SKILL.md`) — a decision table only:

| User intent | Sub-skill |
|---|---|
| "Create a character", "new UGC creator", "make me a persona", update a character's look | `character-creator` |
| "Make an ad image", "product ad", "static creative", "carousel image" | `ad-image` |
| "Make a UGC video", "reel", "video ad", "animate this" | `ugc-video` |

Rule: ad-image and ugc-video **require** a character and/or product to exist first. If the referenced asset isn't in the library, route to `character-creator` before proceeding — never silently generate a throwaway identity.

## 3. The asset library (shared data model)

On-disk, human-inspectable, versioned. Both output skills read from here by ID.

### character.json
```json
{
  "id": "maya-01",
  "name": "Maya",
  "persona": "mid-20s, warm, girl-next-door, casual-chic, soft makeup",
  "appearance": {
    "descriptor": "canonical text description injected into every prompt",
    "wardrobe_default": "oversized cream knit, gold hoops",
    "negative": "no heavy makeup, no tattoos, no sunglasses"
  },
  "refs": ["refs/face_front.png", "refs/face_3q.png", "refs/body_full.png"],
  "seed_prompt": "prompt used to originate her",
  "created_from": "generate | upload",
  "version": 1
}
```

### product.json
```json
{
  "id": "elunoire-heart-mug",
  "name": "Personalized Heart Mug",
  "descriptor": "white ceramic mug, printed photo + name on front",
  "refs": ["refs/front.png", "refs/angle.png"],
  "constraints": "print area must stay legible and undistorted; keep true white"
}
```

**Character origination** (character-creator) produces the `refs/` set: a mini turnaround — front, 3⁄4, profile, full body — from one seed, so the identity can later be composited or animated from any angle. Store the exact `seed_prompt` so the character can be extended (new wardrobe, new expression) while staying on-model.

## 4. Skill: character-creator

**Goal:** produce or update a locked Character (or Product) asset.

Flow:
1. **Interview** the persona (reuse the `content-grill-me` interview instinct — niche, age, vibe, wardrobe, the account's angle). For products, ask for real photos to upload.
2. **Originate** — `generate_reference()` builds the multi-angle ref set from one seed prompt, OR ingest uploaded photos (real product, real person).
3. **Lock & write** — save `refs/` + manifest via `assets.py`. Assign an ID.
4. **Review loop** — show the ref sheet, let the user reject/regen individual angles before locking. Bump `version` on edits.

Powerful bits worth building: a **turnaround ref set** (not one image) so downstream compositing has angles to work from; a **canonical descriptor** that's injected verbatim into every future prompt; and **versioning** so a character can evolve (seasonal wardrobe) without becoming a different person.

## 5. Skill: ad-image

**Goal:** static ad creatives — more powerful than plain text-to-image because they composite the *real* product and/or a *consistent* character, then overlay controlled copy.

Two modes:
- **Character-led (UGC style):** Maya holding/using the product — feels like an organic post.
- **Product-hero (studio):** clean product on a set, no character.

Job schema:
```json
{
  "meta": {
    "brand": "EluNoire",
    "product": "elunoire-heart-mug",
    "character": "maya-01",
    "angle": "Valentine's gift",
    "formats": ["1:1", "4:5", "9:16"]
  },
  "variants": [
    {
      "id": "v1",
      "concept": "Maya surprised, holding mug, morning light",
      "keyframe": {
        "prompt": "...",
        "use_refs": ["maya-01", "elunoire-heart-mug"],
        "negative": "..."
      },
      "overlay": { "headline": "...", "subhead": "...", "cta": "Shop now" }
    }
  ]
}
```

`run_ad.py`:
1. `composite()` the referenced assets into each variant's scene (Nano-Banana).
2. Export each requested format (crop/reframe per aspect).
3. **Render text as a separate layer** (PIL / canvas / SVG) — never rely on the model to draw legible copy. Headline/subhead/CTA come from Claude and are burned in deterministically. This is why the ads look "powerful" instead of AI-sloppy.
4. Batch N variants off one concept for Meta testing.

`_shared/ad-layout.md` holds the composition rules: safe zones per format, hierarchy, CTA placement, contrast.

## 6. Skill: ugc-video

**Goal:** the image-first video pipeline. Character + product → keyframes → animated clips → assembled reel.

**Production JSON** (Claude is the planner; emits this, touches no pixels):
```json
{
  "meta": {
    "brand": "EluNoire", "product": "elunoire-heart-mug", "character": "maya-01",
    "angle": "Valentine's gift", "aspect": "9:16",
    "total_duration_s": 30, "variant_id": "a1"
  },
  "audio": { "vo_voice": "warm-female", "music_mood": "cozy-upbeat" },
  "shots": [
    {
      "id": "s1", "beat": "hook", "duration_s": 6,
      "keyframe": { "prompt": "...", "use_refs": ["maya-01"], "negative": "..." },
      "motion":   { "prompt": "...", "camera": "slow push-in", "action": "looks to camera, smiles" },
      "caption": "POV: you forgot Valentine's again",
      "vo": "Okay so I panic-ordered this and..."
    }
  ]
}
```

**Call sequence (DAG — stages 2 & 3 are parallel):**
```
plan   = llm.plan(brief)                       # stage 0
refs   = load(character) + load(product)       # stage 1 — from library, NOT regenerated
frames = [ images.composite(s.keyframe, refs)  for s in plan.shots ]   # stage 2 ∥
clips  = [ video.image_to_video(f, s.motion)   for f,s in zip(frames, plan.shots) ]  # stage 3 ∥
video  = compose.concat(clips) | captions(plan) | audio(vo=tts, music=mood)          # stage 4
```

Non-negotiables baked into `run_video.py`:
- **References locked once, reused every shot** — pulled from the library, never regenerated.
- **Content-address each shot's outputs** (hash of prompt+refs) so re-runs skip finished work and you can regen a single bad shot via `render_shot.py <production> s3` without rebuilding the batch.
- **Optional continuity mode:** feed clip N's last frame as an extra ref into keyframe N+1 for a seamless flow. Off by default — distinct shots usually beat a morph.

**Talking-head caveat:** neither model gives clean lip-sync out of the box. Default to *VO-over-b-roll* UGC (hands, product, lifestyle — no talking face, VO on top); it converts fine and hides the most artifacts. Leave a hook in `_shared/seedance-motion.md` for a future lip-sync/avatar step for talking beats only.

## 7. Provider abstraction (resolve the host question here)

All model calls go behind three thin modules so the host (fal / Replicate / direct API) is a one-file swap and never leaks into the skills.

```python
# providers/images.py
def generate_reference(prompt, angles, aspect) -> list[Path]: ...   # originate a character ref set
def composite(spec, ref_imgs, aspect) -> Path: ...                  # place refs into a new scene

# providers/video.py
def image_to_video(frame, motion, duration_s) -> Path: ...

# providers/llm.py
def plan(brief) -> dict: ...     # → Production / ad job JSON
def copy(context) -> dict: ...   # captions, VO, headlines
```

Host + key read from env (`.env` / plugin config). **Decision #3 below picks which host each maps to.**

## 8. Build order

1. **Provider layer + asset library first.** `providers/*` against your chosen hosts, `assets.py` create/list/get. Nothing works without these.
2. **character-creator.** Prove you can originate a locked, multi-angle character and re-summon it identically. This validates the whole continuity premise before you build anything on top.
3. **ad-image.** Fastest payoff, single-stage (no video), gives testable Meta creatives immediately.
4. **ugc-video.** The full DAG, `run_video.py` + `compose.py`, content-addressed shots.
5. **Router + slash commands + `_shared` prompt-craft.** Wire it together; move hard-won prompt patterns into `_shared/` as you learn them.
6. **Regen ergonomics.** `render_shot.py`, variant batching for A/B.

Ship 1→3 as a usable v0 (characters + ads). Video is the v1.

## 9. Open decisions (resolve before writing the runner)

1. **Grill-first vs brief-first.** Reuse the `content-grill-me` interview to extract briefs (recommended — native to your workflow), or accept a structured brief and go straight to JSON. Applies to all three skills.
2. **Asset storage.** Local `assets/` in-repo (simple) vs object storage (S3/R2) with manifests pointing at URLs (scales, survives across machines). Start local, design manifests URL-agnostic.
3. **Model hosts — the blocker.** Which host serves each model? This determines the `providers/*` implementations:
   - Nano-Banana → fal? Replicate? direct Gemini API?
   - Seedance → fal? Replicate?
   Pick per-model; the abstraction absorbs the difference but the SDK calls have to be written against the real one.
4. **Text-overlay engine for ads.** PIL, SVG→raster, or a headless-browser HTML render (richest layouts, heaviest dep).
5. **Lip-sync.** Ship VO-over-b-roll only for v1, or bolt on an avatar/lip-sync provider for talking beats now?
